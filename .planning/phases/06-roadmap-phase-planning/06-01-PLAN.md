  <behavior>
    RoadmapOrchestrator takes (db: Database.Database, dispatchTool: ToolHandler) and creates its own PlanningStore — exact parallel to ResearchOrchestrator and RequirementsOrchestrator constructor pattern. Store `db` as an instance field alongside `store` (needed for commitRoadmap transaction).

    **Exported interfaces:**

    PhaseDefinition:
      name: string
      goal: string
      requirements: string[]       // REQ-IDs e.g. ["AUTH-01", "AUTH-02"]
      successCriteria: string[]    // 2-5 observable criteria
      phaseOrder: number

    PlanStep:
      id: string
      description: string
      subtasks: string[]
      dependsOn: string[]          // step IDs

    PhasePlan:
      steps: PlanStep[]
      dependencies: string[]       // REQ-IDs or phase names this plan depends on
      acceptanceCriteria: string[]

    **Methods to implement:**

    generateRoadmap(projectId, projectGoal, toolContext): Promise<PhaseDefinition[]>
    - Reads v1 requirements via store.listRequirements(projectId).filter(r => r.tier === "v1")
    - Dispatches single sessions_dispatch task: roadmap-generation soul, 120s timeout
    - Task context includes full v1 reqs as JSON, project goal, grouping rules (same category = same phase, foundation-first ordering, orphans get catch-all phase)
    - Extracts JSON from code block using /```json\s*([\s\S]*?)```/ regex (same pattern as researcher.ts)
    - Returns parsed PhaseDefinition[]

    commitRoadmap(projectId, phases): void
    - Wraps all createPhase() calls in db.transaction() — atomic batch insert (CRITICAL: avoids partial-roadmap corruption)
    - Deletes existing phases for projectId before inserting (DELETE FROM planning_phases WHERE project_id = ?)
    - Calls store.createPhase() for each PhaseDefinition
    - Validates requirements and successCriteria default to [] if agent omitted them (exactOptionalPropertyTypes guard)
    - NOTE: uses this.db directly for the transaction wrapper; no external db parameter needed

    validateCoverage(projectId, phases): { covered: boolean; missing: string[] }
    - Reads v1 reqs from store.listRequirements(projectId).filter(v1).map(r => r.reqId)
    - Builds Set from phases.flatMap(p => p.requirements)
    - Returns { covered: v1ReqIds.every(id => coveredSet.has(id)), missing: v1ReqIds.filter(id => !coveredSet.has(id)) }

    validateCoverageFromDb(projectId): { covered: boolean; missing: string[] }
    - Reads current persisted phases from store.listPhases(projectId)
    - Reads v1 reqs from store.listRequirements(projectId).filter(v1)
    - Builds coverage set from phases and returns same { covered, missing } shape as validateCoverage
    - Used by the commit action in plan_roadmap tool where no in-memory phases array exists

    listPhases(projectId): PlanningPhase[]
    - Returns store.listPhases(projectId) — thin delegation, sorted by phase_order ascending
    - Required by plan_roadmap tool's adjust_phase and commit actions

    adjustPhase(projectId, adjustment, opts): void
    - opts: { phaseName?: string; requirements?: string[]; newName?: string; newGoal?: string }
    - Gets phase by name: store.listPhases(projectId).find(p => p.name === opts.phaseName) — throws PlanningError if not found
    - Applies structured change: calls store.updatePhase(phaseId, patch) where patch is built from non-undefined opts fields
    - If store.updatePhase does not exist yet on PlanningStore, implement it using db.prepare("UPDATE planning_phases SET ... WHERE id = ?").run(...) directly in this method — support updating: name, goal, requirements (stored as JSON array)
    - The `adjustment` string parameter is the raw freeform user text — log it for diagnostics but the structured opts drive the actual DB update
    - Returns void; callers call listPhases() + formatRoadmapDisplay() separately to get updated display

    planPhase(projectId, phaseId, config, toolContext): Promise<PhasePlan>
    - Gets phase via store.getPhase(phaseId) — throws if not found
    - Calls generatePlanForPhase(phase, depthInstruction, toolContext) — dispatches planning agent
    - If config.plan_check === true: runs checker loop up to MAX_ITERATIONS=3
      - Each iteration: calls checkPlan(phase, plan, toolContext, attempt)
      - If check.pass: break
      - If attempt < 3: calls revisePlan(phase, plan, issues, depthInstruction, toolContext)
      - If attempt === 3 and !pass: log.warn with issues, proceed with best-effort plan
    - Calls store.updatePhasePlan(phaseId, plan) ONLY after checker passes or all iterations exhausted
    - Returns PhasePlan

    Private methods:
    - generatePlanForPhase(phase, depthInstruction, toolContext): Promise<PhasePlan> — dispatches planner agent
    - checkPlan(phase, plan, toolContext, attempt): Promise<{ pass: boolean; issues: string[] }> — dispatches checker agent; on dispatch failure returns { pass: true, issues: [] } (best-effort)
    - revisePlan(phase, plan, issues, depthInstruction, toolContext): Promise<PhasePlan> — dispatches revise agent

    depthToInstruction (exported function or private):
    - "quick" → "Produce a brief plan: 1-3 high-level steps, minimal subtasks."
    - "standard" → "Produce a standard plan: 3-5 steps with key subtasks and acceptance criteria."
    - "comprehensive" → "Produce a detailed plan: 5-10 steps with subtasks, explicit dependencies, full acceptance criteria."

    transitionToPhysicalPlanning(projectId): void
    - store.updateProjectState(projectId, transition("roadmap", "ROADMAP_COMPLETE"))

    transitionToExecution(projectId): void
    - store.updateProjectState(projectId, transition("phase-planning", "PLAN_READY"))

    formatRoadmapDisplay(phases): string
    - Accepts PlanningPhase[] (from store) or PhaseDefinition[] — both have name, goal, requirements, successCriteria
    - Returns markdown: "## Generated Roadmap\n" then for each phase: "### Phase N: name", "**Goal:** goal", "**Requirements:** IDs", "**Success criteria:**", bullet list of criteria, empty line
    - Ends with: "Adjust anything? (e.g., 'Move AUTH-01 to Phase 1', 'Rename Phase 3 to Integration', 'done' to commit)"

    **Test cases:**

    validateCoverage:
    - covered=true when phases cover all v1 reqs
    - covered=false, missing=[reqId] when one v1 req is absent from all phases
    - covered=true with empty phases if no v1 reqs exist

    validateCoverageFromDb:
    - covered=true when persisted phases cover all v1 reqs (use commitRoadmap to seed phases)
    - covered=false, missing=[reqId] when a v1 req is absent from all persisted phases

    listPhases:
    - returns phases sorted by phase_order ascending
    - returns [] when no phases exist for project

    adjustPhase:
    - renaming a phase updates its name in the DB
    - updating requirements replaces the requirements array for that phase
    - throws PlanningError when phase name not found

    planPhase (plan_check=false):
    - calls generatePlanForPhase exactly once, calls store.updatePhasePlan once, never calls checkPlan

    planPhase (plan_check=true, checker passes on first try):
    - calls generatePlanForPhase once, calls checkPlan once (returns pass=true), stores plan

    planPhase (plan_check=true, checker fails then passes):
    - calls generatePlanForPhase once, checkPlan returns fail, revisePlan called, second checkPlan returns pass, stores revised plan

    planPhase (plan_check=true, checker fails 3 times):
    - generatePlanForPhase + checkPlan(fail) + revisePlan + checkPlan(fail) + revisePlan + checkPlan(fail) → stores plan anyway (best-effort)

    depthToInstruction:
    - "quick" → contains "1-3"
    - "standard" → contains "3-5"
    - "comprehensive" → contains "5-10"
  </behavior>
