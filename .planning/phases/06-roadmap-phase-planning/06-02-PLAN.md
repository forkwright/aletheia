Tool name: `plan_roadmap`
Description: "Manage the roadmap generation and phase planning loop. Generate phases from requirements, adjust the roadmap interactively, commit when ready, then generate depth-calibrated plans for each phase."

**Input schema actions:** `generate`, `adjust_phase`, `commit`, `plan_phases`

Required always: `action: string`, `projectId: string`

Additional input by action:
- `generate`: no additional required fields
- `adjust_phase`: `adjustment: string` (freeform text like "Move AUTH-01 to Phase 1"), `phaseName?: string`, `requirements?: string[]`, `newName?: string`, `newGoal?: string`
- `commit`: no additional required fields
- `plan_phases`: `phaseIds?: string[]` (array of phase IDs to plan; if omitted, plans all phases for the project in phase_order sequence)

**Action implementations:**

`generate` action:
- Read project via `orchestrator.getProject(projectId)` — return error JSON if not found
- Call `roadmapOrchestrator.generateRoadmap(projectId, project.goal ?? "", context)` to get PhaseDefinition[]
- Validate coverage via `roadmapOrchestrator.validateCoverage(projectId, phases)`
- If coverage fails: return JSON `{ error: "Coverage validation failed", missing: result.missing }`
- Read `project.config` (cast to PlanningConfig) to check mode
- If `config.mode === "yolo"` (autonomous mode): call `roadmapOrchestrator.commitRoadmap(projectId, phases, db)` immediately, then call `orchestrator.completeRoadmap(projectId, context.nousId, context.sessionId)` and return `{ committed: true, phaseCount: phases.length, message: "Roadmap auto-committed (autonomous mode)." }`
- If interactive mode: call `roadmapOrchestrator.commitRoadmap(projectId, phases, db)` to persist draft (write-on-generate, survives restart — per RESEARCH.md open question recommendation), then return `{ draft: true, display: roadmapOrchestrator.formatRoadmapDisplay(roadmapOrchestrator.listPhases(projectId)), message: "Review the roadmap above. Adjust with adjust_phase action or say 'done' to commit." }`

NOTE: `db` is not directly available in the tool — pass it via closure. `roadmapOrchestrator` already holds a `PlanningStore`; expose `commitRoadmap` as a method on `RoadmapOrchestrator` that uses its internal `this.store` and `this.db`. See RESEARCH.md Pattern 1 for constructor signature — store `db` as instance field alongside `store`.

`adjust_phase` action:
- Read `adjustment: string` from input
- The agent interprets the user's natural language BEFORE calling this action and passes structured data: `phaseName` (which phase to target), and one of: `requirements` (new requirements array to assign), `newName` (rename), `newGoal` (new goal string)
- Call `roadmapOrchestrator.adjustPhase(projectId, adjustment, { phaseName, requirements, newName, newGoal })` — implement `adjustPhase` on RoadmapOrchestrator as: get phase by name, apply the structured change, call `store.updatePhase(phaseId, changes)` — NOTE: if `store.updatePhase` doesn't exist yet, implement it as a store patch in this task using `db.prepare("UPDATE planning_phases SET ... WHERE id = ?").run(...)` directly in RoadmapOrchestrator. Keep it simple: support renaming phase and updating requirements array.
- After adjustment, re-validate coverage; return updated display
- Return `{ adjusted: true, display: roadmapOrchestrator.formatRoadmapDisplay(roadmapOrchestrator.listPhases(projectId)), coverageGate: result.covered }`

`commit` action:
- Validate coverage one final time via `roadmapOrchestrator.validateCoverage(projectId, roadmapOrchestrator.listPhases(projectId).map(p => p.requirements).flat())`
  - Actually: use `roadmapOrchestrator.validateCoverageFromDb(projectId)` — a new zero-argument method that reads current phases from DB and checks against v1 reqs. Add this to RoadmapOrchestrator (simple: reads from store, builds sets, returns { covered, missing }).
- If not covered: return `{ error: "Coverage gate not met", missing: [...] }`
- Call `orchestrator.completeRoadmap(projectId, context.nousId, context.sessionId)`
- Return `{ committed: true, phaseCount: listPhases result.length, message: "Roadmap committed. Ready to generate phase plans." }`

`plan_phases` action:
- Read phases: if `phaseIds` provided use those; otherwise get all phases via `orchestrator.listPhases(projectId)` sorted by `phase_order` ascending, use all their IDs
- Read project config for depth and plan_check flags
- Loop sequentially (not parallel — PHAS-01): for each phaseId, call `await roadmapOrchestrator.planPhase(projectId, phaseId, config, context)`
- After all phases planned, call `orchestrator.advanceToExecution(projectId, context.nousId, context.sessionId)`
- Return `{ planned: phaseIds.length, message: "All phase plans generated. Ready for execution." }`

**oxlint require-await pattern:** Sync actions (`adjust_phase` read path, `commit` coverage check) that are sync must use `Promise.resolve()` not `async`. Async actions (`generate`, `plan_phases`) use `await`. Follow exact pattern from requirements-tool.ts: entire `execute()` is sync, individual actions return `Promise.resolve(...)` or `await someAsyncFn().then(r => JSON.stringify(r))`. Actually: `execute()` returns `Promise<string>` without `async` keyword — same as requirements-tool.ts.

BUT: `generate` and `plan_phases` are truly async (call `roadmapOrchestrator.generateRoadmap` which is async). So the `execute()` function MUST be async. But then oxlint `require-await` is satisfied by those awaits. Sync branches return `Promise.resolve()`. This matches the pattern — check requirements-tool.ts: execute() is NOT async (returns Promise<string>). The fix: wrap the entire body in `return (async () => { ... })()` so the outer function isn't async but awaits work internally. OR: use the pattern where async branches just call the async method and chain `.then()`. The simplest safe pattern: make `execute()` a regular function that returns `Promise<string>`, and use `if (action === "generate") { return roadmapOrchestrator.generateRoadmap(...).then(phases => JSON.stringify({...})); }` — awaiting happens inside the promise chain, no `async` keyword needed on execute(). Use this pattern.

**completeRoadmap(projectId: string, nousId: string, sessionId: string): string**
```typescript
completeRoadmap(projectId: string, nousId: string, sessionId: string): string {
  this.store.updateProjectState(projectId, transition("roadmap", "ROADMAP_COMPLETE"));
  eventBus.emit("planning:phase-complete", { projectId, nousId, sessionId, phase: "roadmap" });
  log.info(`Roadmap complete for project ${projectId}; advancing to phase-planning`);
  return "Roadmap committed. Starting phase planning.";
}
```

**advanceToExecution(projectId: string, nousId: string, sessionId: string): string**
```typescript
advanceToExecution(projectId: string, nousId: string, sessionId: string): string {
  this.store.updateProjectState(projectId, transition("phase-planning", "PLAN_READY"));
  eventBus.emit("planning:phase-started", { projectId, nousId, sessionId, fromState: "phase-planning", toState: "executing" });
  log.info(`Phase planning complete for project ${projectId}; advancing to executing`);
  return "All phase plans ready. Moving to execution.";
}
```
