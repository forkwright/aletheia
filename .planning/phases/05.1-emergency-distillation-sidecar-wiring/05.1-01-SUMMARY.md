# Phase 5.1 Plan 01: Emergency Distillation Sidecar Wiring Summary

**sidecarUrl wired through RuntimeServices to emergency distillation path, emergency flag added to DistillationOpts and distill:before/distill:after event payloads for end-to-end traceability**

## Performance

## Accomplishments

- Emergency distillation (90% context overflow) now receives sidecarUrl from RuntimeServices, enabling dedup, evolution pre-check, and contradiction invalidation — identical to normal distillation path
- Emergency distillations are now distinguishable end-to-end: log entries carry `{ emergency: true }`, DistillationOpts carries `emergency: true`, and both distill:before and distill:after events include `emergency: true` when opts.emergency is set
- Fixed 8 pre-existing test failures in context.test.ts caused by missing memoryIds/memoryTexts in per-test recallMemories mock overrides
- Added 2 new targeted tests confirming sidecarUrl propagation and emergency flag propagation through emergency distillation path

## Task Commits

1. **Task 1: Wire sidecarUrl through RuntimeServices and emergency distillation path** - `da326e2` (feat)

## Files Created/Modified

- `infrastructure/runtime/src/nous/manager.ts` - Added sidecarUrl conditional spread in buildServices()
- `infrastructure/runtime/src/nous/pipeline/stages/context.ts` - Spread sidecarUrl and added emergency: true to emergency distillSession call and log entries
- `infrastructure/runtime/src/melete/pipeline.ts` - Added emergency?: boolean to DistillationOpts, spread into distill:before and distill:after events
- `infrastructure/runtime/src/nous/pipeline/stages/context.test.ts` - Fixed mock breakage, added 2 new tests

## Decisions Made

- `sidecarUrl` added to `RuntimeServices` rather than `TurnState` — service-level config follows the established pattern of `memoryTarget`, `plugins`, and `watchdog` being on services, not per-turn state
- `emergency: true` is always set (non-conditional) in the context.ts call site because that call IS the emergency path. In pipeline.ts, the event emits use conditional spread `...(opts.emergency ? { emergency: true } : {})` to ensure normal distills never emit `emergency: undefined`
- Test assertion uses `mock.calls[N][4]` direct access because `expect.anything()` in vitest excludes `undefined`, and `services.router` is undefined in the test's makeServices helper

## Deviations from Plan

None - plan executed exactly as written. The test assertion style differed from the plan example (used `mock.calls` access instead of `toHaveBeenCalledWith`) to work around the undefined router in test services, but this is an implementation detail within the same task scope.

## Issues Encountered

- `expect.anything()` in vitest excludes `null` and `undefined` — the test helper `makeServices` does not set a `router` field, making services.router undefined. The initial assertion using `expect.anything()` for the router argument position failed. Fixed by reading mock.calls directly and asserting only on the opts argument (position 4).

None - no external service configuration required.
