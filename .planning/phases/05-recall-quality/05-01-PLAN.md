Output: Memory IDs and texts threaded from recall through finalize, Jaccard usage-detection heuristic (>=0.25 threshold) calling /evolution/reinforce only for used memories, exponential decay formula in evolution.py replacing linear penalty in _apply_confidence_weight.

**recall.ts changes:**
- Add `memoryIds: string[]` to the `RecallResult` interface (new field alongside block, count, durationMs, tokens)
- Add `memoryTexts: Map<string, string>` to `RecallResult` — maps each memory ID to its text content. This is required so `finalize.ts` can run the Jaccard usage-detection heuristic (per locked decision: reinforce only when the agent *uses* a recalled memory, not on every recall hit).
- After the dedup/select phase in `recallMemories()`, collect IDs: `const memoryIds = deduped.map(h => h.id).filter((id): id is string => !!id);`
- Build the memoryTexts map: `const memoryTexts = new Map(deduped.filter(h => h.id).map(h => [h.id!, h.memory]));`
- Include both `memoryIds` and `memoryTexts` in the returned RecallResult object
- The `MemoryHit` interface already has an optional `id` field — no change needed there

**types.ts changes:**
- Add `recalledMemoryIds?: string[]` to the `RuntimeServices` or `TurnState` type (whichever carries recall results to finalize). Check the exact flow: context.ts calls recallMemories and the result flows through pipeline stages. Thread memoryIds into whatever structure finalize.ts reads from.
- Add `recalledMemoryTexts?: Map<string, string>` alongside — the Jaccard heuristic needs the memory text to compare against the response.

**finalize.ts changes:**
- Add a `tokenJaccardOverlap(a: string, b: string): number` helper function (module-level):
  - Tokenize both strings: lowercase, split on whitespace, filter tokens shorter than 3 chars
  - Compute Jaccard: `intersection.size / union.size` (return 0 if union is empty)
  - This detects whether a memory influenced the response per locked user decision: "Reinforce only when the agent uses a recalled memory in its response — not on every recall hit."
- Add a `reinforceUsedMemories` function (module-level, not inside finalize):
  - Takes `memoryIds: string[]`, `memoryTexts: Map<string, string>`, `responseText: string`, `sidecarUrl: string`, `nousId: string`
  - For each memoryId, get the memory text from `memoryTexts.get(memoryId)`
  - Compute `tokenJaccardOverlap(memoryText, responseText)` — if overlap >= 0.25, this memory was "used" (threshold 0.25 is permissive per RESEARCH.md recommendation: false positives are cheap, false negatives hurt)
  - Only reinforce memories that pass the threshold
  - For each qualifying memory, POST to `${sidecarUrl}/evolution/reinforce` with `{memory_id, user_id: nousId}` — fire-and-forget, non-blocking
  - Wrap the entire function in try/catch that logs warn and returns (never throws)
- At the end of the finalize stage, after saving outcome: if `recalledMemoryIds` has entries and `outcome.text.length > 100`, call `void reinforceUsedMemories(recalledMemoryIds, recalledMemoryTexts, outcome.text, sidecarUrl, nousId)` — fire-and-forget using `void` prefix

**Test updates:**
- recall.test.ts: Add test that RecallResult includes both memoryIds array and memoryTexts Map from deduped hits
- finalize.test.ts: Add tests for tokenJaccardOverlap:
  - Identical strings → 1.0
  - No shared tokens → 0.0
  - Partial overlap with known expected value
  - Short tokens (<3 chars) are excluded from computation
- finalize.test.ts: Add tests for reinforcement with usage detection:
  - When a recalled memory's text has Jaccard overlap >= 0.25 with response, fetch is called with /evolution/reinforce for that memory ID
  - When a recalled memory's text has Jaccard overlap < 0.25 with response, fetch is NOT called for that memory ID (memory was not used)
  - When no recalled memory IDs, fetch is not called for reinforcement
  - When fetch rejects, finalize completes without error (fire-and-forget resilience)
  - Mixed scenario: 3 memories recalled, only 1 has overlap >= 0.25 — only that 1 is reinforced

Use `createLogger("nous:reinforcement")` for the reinforcement function logger.

**evolution.py changes:**
- Add `import math` at top if not present
- Add an `exponential_decay_penalty(days_inactive: float, lambda_: float = 0.05) -> float` function:
  - Returns `math.exp(-lambda_ * days_inactive)` — score multiplier in [0, 1]
  - lambda=0.05 gives ~14-day half-life: 7 days → 0.70, 14 days → 0.50, 30 days → 0.22, 90 days → 0.01
- Modify the `/evolution/decay` endpoint: ensure it stores `last_accessed` timestamp in the MemoryAccess Neo4j node (it already stores `last_accessed` in the reinforce endpoint — verify the decay endpoint does NOT overwrite `last_accessed`). The decay endpoint should only write `decay_count` and `last_decayed` — never touch `last_accessed`.
- Export `exponential_decay_penalty` for use in routes.py

**routes.py changes (in _apply_confidence_weight or equivalent scoring function):**
- Replace the linear penalty (`decays * 0.02`) with exponential decay:
  - Read `last_accessed` from the MemoryAccess Neo4j node (already queried in this function)
  - Compute `days_inactive = (now - last_accessed).days` (or fractional days)
  - Apply `score *= exponential_decay_penalty(days_inactive)`
  - If `last_accessed` is not set (no MemoryAccess node), apply no penalty (score unchanged) — new memories have full salience until first access
- Import `exponential_decay_penalty` from `evolution` module

  - 30 days inactive → multiplier ~0.22
  - 90 days inactive → multiplier ~0.01
- Test that /evolution/decay endpoint does NOT modify `last_accessed` field
- Test that /evolution/reinforce endpoint DOES update `last_accessed` to current time

Do NOT add numpy/scipy — use `math.exp` only (project pattern from Phase 4).
Do NOT modify the reinforce endpoint behavior — only verify it.
