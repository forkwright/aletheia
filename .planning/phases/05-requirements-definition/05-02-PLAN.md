Constructor: `constructor(db: Database.Database)` — creates `new PlanningStore(db)` internally. No dispatchTool needed (requirements logic is synchronous).

**Interfaces to export:**

```typescript
export interface FeatureProposal {
  name: string;
  description: string;
  isTableStakes: boolean;
  proposedTier: "v1" | "v2" | "out-of-scope";
  proposedRationale?: string;   // only meaningful when proposedTier is "out-of-scope"
}

export interface CategoryProposal {
  category: string;      // Short code: "AUTH", "STOR", "API" (2-6 uppercase chars)
  categoryName: string;  // Human readable: "Authentication", "Data Storage"
  tableStakes: FeatureProposal[];
  differentiators: FeatureProposal[];
}

export interface ScopingDecision {
  name: string;
  tier: "v1" | "v2" | "out-of-scope";
  rationale?: string;  // required when tier is "out-of-scope"
}
```

**Methods to implement:**

1. `getSynthesis(projectId: string): string | null`
   - Calls `this.store.listResearch(projectId)`, finds row where `dimension === "synthesis"`
   - Returns `row.content ?? null`
   - Returns null if no synthesis (research was skipped)

2. `formatCategoryPresentation(category: CategoryProposal): string`
   - Returns markdown string for agent to display
   - Format: `## {categoryName} ({category})\n\n**Table stakes** (users expect these):\n...`
   - Each feature line: `- **{name}**: {description} → proposed: **{proposedTier}**`
   - Ends with: `Confirm these proposals or adjust (e.g., 'move the second one to v2', 'make all v1'):`
   - No trailing newline

3. `persistCategory(projectId: string, category: CategoryProposal, decisions: ScopingDecision[]): void`
   - Queries existing requirements for this category: `store.listRequirements(projectId).filter(r => r.category === category.category)`
   - Finds MAX number already used: parse `-NN` suffix from reqId, default to 0 if none exist
   - For each decision in decisions array (in order):
     - Generate reqId: `${category.category}-${String(nextNum).padStart(2, "0")}`; increment nextNum
     - Build description from matching FeatureProposal (tableStakes + differentiators combined)
     - Enforce user-centric testability: description must start with "User can" or contain an observable behavior (if FeatureProposal description is vague/implementation-focused, prefix with "User can " as a minimal fix)
     - Call `store.createRequirement({ projectId, reqId, description, category: category.category, tier: decision.tier, rationale: decision.tier === "out-of-scope" ? (decision.rationale ?? null) : null })`
   - Log: `log.info("Persisted N requirements for category CAT")`

4. `updateRequirement(projectId: string, reqId: string, updates: { tier?: "v1" | "v2" | "out-of-scope"; rationale?: string | null }): void`
   - Finds requirement by reqId in listRequirements(projectId)
   - Calls store.updateRequirement(row.id, updates) — uses DB row id, not reqId
   - Throws PlanningError with PLANNING_REQUIREMENT_NOT_FOUND if reqId not found in this project

5. `validateCoverage(projectId: string, presentedCategories: string[]): boolean`
   - Two conditions (both must be true per CONTEXT.md):
     1. At least 1 requirement with tier === "v1" exists
     2. Every category in presentedCategories has at least 1 requirement row
   - Returns false if either condition fails

6. `transitionToRoadmap(projectId: string): void`
   - Calls `this.store.updateProjectState(projectId, transition("requirements", "REQUIREMENTS_COMPLETE"))`

**Key constraints from CONTEXT.md (locked):**
- Category code is uppercase abbreviation (2-6 chars) derived from category name
- Numbers start at 01 within each category, auto-incremented from existing max
- REQ-IDs are immutable once created — only tier/rationale are mutable via updateRequirement
- Coverage gate: 1+ v1 exists AND all presented categories covered (not just some)

**Anti-patterns to avoid (from RESEARCH.md):**
- Never hardcode category list — categories come from CategoryProposal parameter
- Never use separate categories table — category is a TEXT column on planning_requirements
- Never call transition() directly from tool layer — RequirementsOrchestrator owns all transitions

**Create `infrastructure/runtime/src/dianoia/requirements-tool.ts`** — mirrors `research-tool.ts` structure.

Export function: `createPlanRequirementsTool(orchestrator: DianoiaOrchestrator, requirementsOrchestrator: RequirementsOrchestrator): ToolHandler`

The tool definition:

```
name: "plan_requirements"
description: "Manage the requirements scoping loop. Present feature categories to the user one at a time with v1/v2/out-of-scope proposals, persist confirmed decisions, handle freeform adjustments, check coverage gate, and advance to roadmap phase when complete."
```

Input schema — `action` enum with 4 values:

- `"present_category"` — accepts `projectId: string` (returns synthesis content for agent to build category presentation from; returns null message when synthesis absent with fallback guidance)
- `"persist_category"` — accepts `projectId: string`, `category: CategoryProposal`, `decisions: ScopingDecision[]` (calls requirementsOrchestrator.persistCategory(); returns coverage status)
- `"update_requirement"` — accepts `projectId: string`, `reqId: string`, `updates: { tier?, rationale? }` (calls requirementsOrchestrator.updateRequirement(); returns updated requirement summary)
- `"check_coverage"` — accepts `projectId: string`, `presentedCategories: string[]` (calls validateCoverage(); returns gate status + counts)
- `"complete"` — accepts `projectId: string`, `presentedCategories: string[]` (calls validateCoverage() first — rejects if gate not met; then calls orchestrator.completeRequirements(); returns confirmation message)

**Tool execute() body pattern from research-tool.ts:**
- Use `await Promise.resolve()` for sync branches if needed to satisfy async interface (or structure with proper await)
- User-facing messages built here, not in orchestrator
- Return `JSON.stringify({ ... })` for all branches
- Wrap in try/catch returning `JSON.stringify({ error: message })` on failure
- All error messages should be user-readable (agent will surface them)

**For the `present_category` action:**
- Call `requirementsOrchestrator.getSynthesis(projectId)`
- If synthesis is null: return message explaining no research was performed, agent should derive categories from project context (CONTEXT.md fallback guidance: use project goal + constraints)
- If synthesis exists: return `{ synthesis: content }` for agent to parse and propose categories from

**For coverage responses:**
- Return `{ covered: boolean, v1Count: number, categoriesCovered: number, categoriesTotal: number, message: string }`

**`orchestrator.ts` — add `completeRequirements()` method:**

```typescript
completeRequirements(projectId: string, nousId: string, sessionId: string): string {
  this.store.updateProjectState(projectId, transition("requirements", "REQUIREMENTS_COMPLETE"));
  eventBus.emit("planning:phase-complete", { projectId, nousId, sessionId, phase: "requirements" });
  log.info(`Requirements complete for project ${projectId}; advancing to roadmap`);
  return "Requirements confirmed. Advancing to roadmap generation.";
}
```

Note: This method is synchronous (same as skipResearch() — no await, satisfies oxlint require-await since it has no async keyword). Emit `"planning:phase-complete"` (confirmed in STATE.md decisions — this event exists in EventName union from Phase 2).
