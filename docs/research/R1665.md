# R1665: Evaluate VeraCrypt Hidden Containers for Instance Data

**Date:** 2026-03-19
**Author:** Research agent
**Status:** Final
**Closes:** #1665

---

## Executive Summary

VeraCrypt hidden volumes are technically sound for their designed use case — manual, interactive protection of sensitive data under direct compulsion — but are a poor fit for aletheia's instance data. The plausible deniability guarantee collapses under automation: any system that mounts a hidden volume programmatically must record the hidden volume's credentials and existence somewhere accessible to the process, which defeats the core claim. Beyond the deniability problem, hidden volumes carry non-trivial operational hazards (outer volume journaling breaks hidden volume integrity, WAL-mode SQLite on the outer volume creates unpredictable writes, NVMe I/O drops 44–53% under AES-XTS), and mount latency with default PBKDF iteration counts ranges from seconds to minutes.

For aletheia's actual threat model — data-at-rest protection against physical disk theft — dm-crypt/LUKS with a keyfile delivers equivalent protection with better automation, lower latency, and simpler operational hygiene.

**Recommendation: Skip VeraCrypt hidden containers.** Evaluate dm-crypt/LUKS as the implementation path for data-at-rest encryption if that feature is prioritised.

## 1. Aletheia Instance Data Profile

Understanding what needs protection frames the threat model analysis.

### 1.1 Data Inventory

| Data type | Location | Format | Size profile |
|---|---|---|---|
| Session messages | `instance/data/sessions.db` | SQLite WAL | Grows with usage |
| Knowledge graph | `instance/data/engine/` | CozoDB + fjall (LSM) | Grows with memory operations |
| Vector embeddings | `instance/data/engine/` | HNSW index files | ~384 dims × N facts |
| Agent workspaces | `instance/nous/{id}/` | Markdown, TOML | Small |
| Configuration | `instance/config/aletheia.toml` | TOML | Small |
| Credentials | `instance/config/credentials/` | Various | Tiny |
| Logs | `instance/logs/` | Structured text | Rotated |
| Session archives | `instance/data/archive/` | JSON | Append-only |
| Signal state | `instance/signal/` | signal-cli format | Small–medium |

### 1.2 Current Encryption State

Aletheia already encrypts sensitive configuration values (API keys, tokens, signing keys) using XChaCha20Poly1305 with a 32-byte master key stored at `~/.config/aletheia/master.key`. The `enc:` prefix marks ciphertext values in `aletheia.toml`. This is implemented in `crates/taxis/src/encrypt.rs`.

Session content, knowledge graph data, vector embeddings, and workspace files are stored **plaintext**. These are the files R1665 proposes to protect.

### 1.3 Write Characteristics

- **SQLite** (`sessions.db`): WAL mode, `synchronous=NORMAL` by default. Writes are sequential to the WAL file; reads are random across the main database. Background checkpoints periodically merge WAL into the main database file.
- **fjall/LSM** (knowledge engine): write-buffered with LZ4-compressed SSTables. Background compaction threads rewrite SSTables in sorted order, producing significant sequential write amplification (10–30× in typical LSM workloads). Random reads for fact retrieval.
- **HNSW index**: updated on every fact insertion; primarily sequential during batch indexing.

## 2. VeraCrypt Hidden Volume Mechanics

### 2.1 Container Layout

A VeraCrypt hidden volume shares a single file container or block device with an outer (decoy) volume. The header region layout for a file container:

| Offset | Contents |
|---|---|
| 0–65,535 | Primary outer volume header |
| 65,536–131,071 | Hidden volume header (appears random without hidden password) |
| 131,072 → end−131,072 | Outer volume data |
| end−131,072 → end−65,536 | Backup hidden volume header |
| end−65,536 → end | Backup outer volume header |

The hidden volume occupies the tail of the container, growing backwards. Both headers are encrypted; without the correct password, the hidden volume header is cryptographically indistinguishable from random noise. The free space of the outer volume — where the hidden volume resides — is filled with random data at container creation time.

This is the basis of plausible deniability: the holder can reveal the outer password and a decoy filesystem, while the hidden volume's existence cannot be *proven* cryptographically.

### 2.2 Key Derivation

Default parameters for non-system volumes use **PBKDF2-SHA-512 with 500,000 iterations**. This is intentionally slow to resist brute force. The side effect is that each mount operation is computationally expensive. Observed mount times:

- Modern desktop (i7/Ryzen with AES-NI): 3–8 seconds
- Server without AES-NI: 20–60 seconds
- Large containers (≥1TB) on some systems: reported up to 6.75 minutes

The PIM (Personal Iterations Multiplier) parameter modifies the iteration formula: `iterations = 15,000 + (PIM × 1,000)`. PIM=0 means 500,000 iterations. PIM=1 means 16,000 iterations. Reducing PIM to accelerate mounts proportionally weakens brute-force resistance.

### 2.3 Outer Volume Write-Protection

When the outer volume is mounted with `--protect-hidden=yes`, VeraCrypt decrypts the hidden volume header in RAM to determine the hidden area's boundaries, then blocks any write that would land there. On the first blocked write, the **entire outer volume becomes read-only** until unmount. This is a binary state: either all outer writes succeed (and may destroy the hidden volume if the protection is not active), or the outer volume is write-locked.

This design means the outer and hidden volumes cannot be used concurrently in any meaningful read-write capacity.

## 3. Linux Integration

### 3.1 Package Availability

VeraCrypt is not in Ubuntu or Debian main repositories. Installation requires:
- Ubuntu: third-party PPA `ppa:unit193/encryption`
- Arch Linux: `community/veracrypt` package
- Compiling from source as fallback

**cryptsetup** (standard on all major distributions since version 1.6) can *open* VeraCrypt containers natively via `--type tcrypt --veracrypt` without the VeraCrypt package. Container *creation* still requires the `veracrypt` binary.

### 3.2 Device Mapper Path

VeraCrypt on Linux uses kernel device mapper (same as dm-crypt/LUKS), placing the block device at `/dev/mapper/<name>`. This is the full kernel-level I/O path, not FUSE. A typical mount sequence:

```bash
# Open hidden volume via cryptsetup (no veracrypt package required)
cryptsetup --type tcrypt --veracrypt --tcrypt-hidden \
  --key-file /path/to/keyfile \
  open /path/to/container.vc aletheia_data

# Mount filesystem on the device mapper node
mount /dev/mapper/aletheia_data /var/lib/aletheia/data
```

### 3.3 Systemd and crypttab Integration

cryptsetup's tcrypt mode integrates with `/etc/crypttab`:

```
# /etc/crypttab
aletheia_data  /path/to/container.vc  /etc/aletheia.keyfile  tcrypt-veracrypt,tcrypt-hidden,tcrypt-keyfile=/etc/aletheia.keyfile
```

```
# /etc/fstab
/dev/mapper/aletheia_data  /var/lib/aletheia/data  ext4  defaults  0  0
```

**Critical finding:** The `tcrypt-hidden` option in `/etc/crypttab` stores plaintext evidence that a hidden volume exists in a system configuration file readable by root. This completely defeats plausible deniability. Any automated mount of a hidden volume necessarily records its existence in a location that system administrators, forensic investigators, or attackers with root access can read.

## 4. Performance Impact

### 4.1 Benchmark Data

No published benchmarks combine VeraCrypt specifically with SQLite WAL workloads. The applicable data comes from dm-crypt AES-XTS benchmarks (same cipher, same kernel path as VeraCrypt in cryptsetup tcrypt mode) and general VeraCrypt I/O reports.

**VeraCrypt on NVMe (AES-XTS, loop device via veracrypt binary):**
A tracked GitHub issue (veracrypt/VeraCrypt#136) reported sequential read speeds dropping to approximately **20% of unencrypted throughput** on a Samsung 960 EVO. The root cause appears to be 512-byte sector handling in the VeraCrypt loop layer rather than the cipher itself. Using cryptsetup's tcrypt mode bypasses this and aligns with dm-crypt benchmarks.

**dm-crypt AES-XTS (2023 Sovereign Cloud Stack benchmark with fio):**

| Storage type | Operation | Overhead |
|---|---|---|
| SAS HDD (10K RPM) | Sequential read | −53% |
| SAS HDD (10K RPM) | Sequential write | −79% |
| SAS HDD (10K RPM) | Random 4K read IOPS | −24% |
| SAS HDD (10K RPM) | Random 4K write IOPS | −43% |
| NVMe SSD (Samsung PM983) | Sequential operations | −83% |
| NVMe SSD | Random 4K read IOPS | −44% |
| NVMe SSD | Random 4K write IOPS | −53% |

Source: Sovereign Cloud Stack, "Performance Impact of Disk Encryption Using LUKS," 2023.

**CPU-bound saturation note:** At NVMe speeds, the dominant bottleneck is CPU throughput for AES operations, not the cipher algorithm. Systems with AES-NI hardware acceleration (Intel Westmere+, AMD Bulldozer+) recover substantially. The VeraCrypt developer has stated less than 3% overhead for sequential AES on AES-NI hardware; the gap at NVMe speeds reflects the fact that NVMe throughput exceeds what a single AES-NI pipeline can sustain.

### 4.2 SQLite WAL Mode on Encrypted Loop Devices

SQLite WAL mode's primary write pattern is sequential appends to `<db>-wal`. Checkpoints merge the WAL into the main database with sequential writes. Query reads are random I/O.

Key findings:
- **`synchronous=NORMAL`** (aletheia's current default): fsyncs occur only at checkpoint, not per-transaction. No change in crash semantics on encrypted block devices.
- **`synchronous=FULL`**: required for power-loss durability. Each fsync traverses the full loop-device → dm-crypt → underlying disk stack. The ~44–53% random IOPS overhead applies to every checkpoint fsync.
- **WAL file location matters**: if the WAL and main database are on the encrypted volume, the WAL's sequential appends incur sequential write overhead (~53–79% on HDD, lower on NVMe with AES-NI). If the volume is on an NVMe SSD with AES-NI, real-world query latency impact for typical aletheia workloads (single-user, moderate query rate) should be measurable but sub-second.

### 4.3 Fjall/LSM on Encrypted Containers

fjall's LSM architecture creates a compaction-dominant write profile:

- **Memtable flushes**: sequential writes of compacted SSTable files — full AES-XTS overhead on write throughput.
- **Compaction**: reads SSTables, merges, and rewrites. Each read and write passes through the encryption layer. With LSM write amplification of 10–30× and ~43–53% encrypted write IOPS overhead, effective compaction throughput is roughly half to two-thirds of unencrypted rates.
- **Bloom filter lookups**: random reads per query — subject to ~44% random read IOPS overhead.
- **No correctness issues**: LSM trees have no additional crash-safety or consistency problems on encrypted block devices beyond the standard `O_SYNC`/fsync considerations.

For aletheia's knowledge engine workload (fact ingestion, vector search, Datalog queries), the primary observable impact would be longer compaction windows and potentially higher compaction stall frequency under sustained write load (e.g., bulk memory ingestion). Interactive query latency is dominated by bloom filter reads and point lookups, which would see the ~44% random read IOPS overhead — measurable but unlikely to cross a user-perceptible threshold for single-agent use.

## 5. Automation Potential

### 5.1 Keyfile Authentication

VeraCrypt supports keyfile-only authentication (no passphrase). The keyfile content is processed as additional key material through PBKDF. For automated mounting:

```bash
cryptsetup --type tcrypt --veracrypt \
  --key-file /etc/aletheia.keyfile \
  open /var/lib/aletheia/container.vc aletheia_data
```

The keyfile must be readable by the process performing the mount. On a systemd-managed service, this means the keyfile is accessible to anyone with root access to the running system.

### 5.2 Fundamental Automation Contradiction

Hidden volumes require plausible deniability to be meaningful: the user must be able to credibly deny the hidden volume exists. Automated mounting requires:

1. The hidden volume's credentials (keyfile path or passphrase) to be available at mount time
2. A record of which volume type to mount (`tcrypt-hidden` vs outer)
3. A mount script, systemd unit, or crypttab entry that references the hidden volume

Each of these is discoverable by an adversary with access to the system. The moment an automated system uses the hidden volume, evidence of its existence appears in:
- `/etc/crypttab` (`tcrypt-hidden` flag)
- `/etc/fstab` (mount point)
- systemd journal (`ExecStart` command or cryptsetup log)
- Process table during mount
- bash history if mounted interactively at any point
- The keyfile's mere existence and its permissions

This is not a configuration problem — it is a structural incompatibility. Hidden volumes are designed for **manual, interactive, adversarial** contexts. Automated agent systems are the opposite: they depend on predictable, scripted, logged behaviour.

### 5.3 Mount Latency in Production Context

With default PBKDF parameters, each aletheia startup would block for 3–60+ seconds waiting for the container to open. For a background service that is expected to restart cleanly, this is operationally disruptive. PIM reduction trades brute-force resistance for speed; a PIM=485 reduces to approximately 500K iterations (the default), while a PIM=0 leaves the standard iteration count. Lowering PIM to, say, 10 reduces to 25,000 iterations and mount time to under a second but with meaningfully weaker key derivation.

## 6. Plausible Deniability: Evidence of Failure Modes

This section documents the attacks that compromise the hidden volume guarantee. These are particularly relevant because aletheia is an automated system that writes continuously.

### 6.1 Filesystem Journaling on Outer Volume

Journaling filesystems (ext4, XFS, NTFS) write metadata to a journal at unpredictable offsets. The journal can write to regions the filesystem considers free — which is exactly where the hidden volume resides. Using ext4 on the outer volume with default journaling creates a genuine risk of:

- Silent destruction of hidden volume data
- Journal entries that land in the hidden volume area, creating forensically detectable write patterns

**VeraCrypt's own documentation recommends FAT32 for the outer volume** to avoid journaling. FAT32 is unsuitable for Linux system directories and does not support files larger than 4GB.

### 6.2 Snapshot-Based Forensics

If the host uses LVM snapshots, btrfs/ZFS CoW snapshots, or VM checkpoints, two snapshots bracketing a hidden volume session will show writes in regions the outer filesystem believes are empty. This is a documented forensic attack with published methodology (Spooren et al., "Defeating Plausible Deniability," Springer, 2017). For any aletheia deployment on a virtualized host or a host with LVM, this attack is feasible.

### 6.3 RAM Key Extraction

While the hidden volume is mounted, the AES-XTS master key resides in kernel memory (in the dm-crypt key slot). VeraCrypt's RAM key encryption feature — which obfuscates keys in memory — is **Windows-only as of version 1.26**. On Linux, an adversary with root access (or physical access for a cold-boot attack) can extract the active key from `/proc/kcore` or kernel memory dumps. Tools like Passware Kit Forensic automate this. On a running aletheia instance, the hidden volume key is recoverable by root.

### 6.4 Wear-Leveling

SSD controllers remap logical blocks for wear distribution. Writes to the hidden volume area create SSD-internal records that persist even if the outer volume is later written. Physical NAND analysis (chip-off forensics) can in principle recover traces that differ from a genuinely empty outer volume. This requires sophisticated adversaries with hardware capabilities; it is relevant only in high-stakes forensic contexts.

### 6.5 TRIM/Discard

TRIM commands inform the SSD which logical blocks are unused. The outer filesystem considers the hidden volume area free and may issue TRIM for those blocks. VeraCrypt disables TRIM by default for non-system volumes, but this must be verified in deployment. A TRIM'd hidden volume on an SSD leaves the SSD firmware with records of which areas were "emptied" — potentially recoverable via vendor diagnostic tools.

### 6.6 What Actually Works

Hidden volumes provide strong protection in a narrow scenario: a powered-off disk, no prior snapshots, no RAM forensics opportunity, HDD media (no wear-leveling), and an adversary who must rely solely on cryptographic analysis. For a self-hosted AI agent system that runs continuously and writes to its data store on every interaction, exactly none of these conditions hold.

## 7. Threat Model Fit

### 7.1 What Aletheia's Data Needs to Be Protected Against

The realistic threat model for a self-hosted AI agent:

| Threat | Likelihood | Data value |
|---|---|---|
| Physical disk theft (powered off) | Low–medium | High: conversation history, API keys, personal knowledge graph |
| Unauthorized filesystem access (root compromise) | Low | High: same data as above |
| Legal compulsion to produce data | Very low for most operators | Varies |
| Snapshot/backup exfiltration | Low–medium | High |

### 7.2 What Hidden Volumes Provide vs. What Is Needed

| Property | Hidden volumes | LUKS | Aletheia's need |
|---|---|---|---|
| Data-at-rest encryption (powered off) | Yes | Yes | **Required** |
| Plausible deniability under compulsion | Yes (conditionally) | No | **Not required** for most deployments |
| Automation compatibility | Defeats PD | Full | **Required** |
| Journaling filesystem support | Outer: no; inner: yes | Yes | **Required** (ext4 default) |
| TRIM/SSD support | No | Yes (opt-in) | Desirable |
| Mount time with keyfile | 3–60+ seconds | 1–3 seconds | **Fast startup required** |
| Systemd/crypttab integration | Workarounds required | Native | **Required** |
| VeraCrypt binary dependency | Yes (for creation) | No | Undesirable |
| Active maintenance/audit | Yes (2016, 2020 audits) | Yes (ongoing) | Required |

### 7.3 The Gap

The one capability hidden volumes add — plausible deniability — is rendered unusable by aletheia's architecture. aletheia is a daemon that starts at boot, mounts its storage programmatically, writes continuously, and stores all configuration in `/etc/`. There is no scenario in which an automated aletheia instance can plausibly deny its hidden volume to a forensically competent adversary who has access to the system's configuration files.

Conversely, the one thing aletheia needs — reliable, automated, fast data-at-rest encryption — is better served by LUKS.

## 8. Alternative: dm-crypt/LUKS

### 8.1 Capabilities

dm-crypt/LUKS2 provides block-level encryption using the kernel's device mapper. It is the standard Linux data-at-rest encryption stack, used in full-disk encryption on Ubuntu, Fedora, Debian, and most Linux distributions.

Key properties:
- **Keyfile support**: native. A 4096-byte random keyfile stored at a protected path unlocks the volume without a passphrase.
- **crypttab integration**: first-class. No workarounds or third-party packages.
- **Mount time**: 1–3 seconds with Argon2id KDF (LUKS2 default), which is faster than VeraCrypt's PBKDF2-SHA-512 at equivalent security margin.
- **Filesystem**: any Linux filesystem (ext4, XFS, btrfs) works without restriction.
- **TRIM**: supported with opt-in `discard` mount flag (security trade-off: reveals which blocks are in use, but does not expose data).
- **No external dependency**: cryptsetup is installed by default on all major Linux distributions.

### 8.2 Aletheia-Specific Configuration Sketch

A LUKS2 container for `instance/data/`:

```
# /etc/crypttab
aletheia_data  /var/lib/aletheia/data.luks  /etc/aletheia.keyfile  luks,key-slot=0

# /etc/fstab
/dev/mapper/aletheia_data  /var/lib/aletheia/data  ext4  defaults,noatime  0  0
```

The keyfile at `/etc/aletheia.keyfile` would be readable only by root (`chmod 600`), providing the same security as the existing `~/.config/aletheia/master.key` pattern already used for config encryption.

This is a direct extension of aletheia's current security model (XChaCha20Poly1305 for config, LUKS AES-XTS for bulk data) without architectural change.

### 8.3 Performance Comparison

On a typical modern host with AES-NI:

| Workload | Unencrypted | LUKS AES-XTS | VeraCrypt AES-XTS |
|---|---|---|---|
| Sequential read (HDD) | Baseline | −53% | −53% (same cipher) |
| Sequential write (HDD) | Baseline | −79% | −79% (same cipher) |
| Random 4K read IOPS (NVMe) | Baseline | −44% | −44% (same cipher) |
| Random 4K write IOPS (NVMe) | Baseline | −53% | −44%–20%* |
| Mount time (keyfile) | — | 1–3 seconds | 3–60+ seconds |

\* VeraCrypt via its own binary has additional loop-device overhead reported at up to 80% throughput loss on some NVMe configurations; cryptsetup tcrypt mode matches dm-crypt performance.

At AES-NI hardware acceleration: sequential overhead approaches the sub-3% range the VeraCrypt developer cited for AES-NI-capable CPUs. For aletheia's interactive single-user workload (not sustained multi-queue I/O), per-operation latency matters more than throughput; the 44% random IOPS overhead translates to roughly 1.75× latency per encrypted random I/O operation.

## 10. Recommendation

**Skip VeraCrypt hidden containers.**

The plausible deniability property that distinguishes VeraCrypt hidden volumes from standard encrypted containers is structurally incompatible with automated operation. An aletheia instance cannot credibly deny its hidden volume to any adversary with access to the system's configuration files, journals, or process table — which are precisely the adversaries the plausible deniability mechanism is intended to protect against.

The remaining value proposition — data-at-rest encryption for the `instance/` directory — is better served by dm-crypt/LUKS, which:

1. Requires no additional packages
2. Has native systemd/crypttab integration
3. Mounts in 1–3 seconds vs. VeraCrypt's 3–60+ seconds
4. Imposes no filesystem restrictions on encrypted content (any Linux filesystem works)
5. Does not require managing an outer/hidden volume split
6. Delivers identical AES-XTS performance when accessed via cryptsetup

If data-at-rest encryption for `instance/data/` is prioritised, the implementation path is a LUKS2 container with a keyfile, extending the existing pattern already used for `~/.config/aletheia/master.key`. The `taxis` crate's encryption infrastructure is a natural home for the mount lifecycle hooks or documentation.

VeraCrypt's full-volume encryption (without hidden volumes) is a legitimate data-at-rest option if portability across platforms (Windows/macOS) is ever required and the added operational complexity is acceptable. That is a narrower use case not relevant to the current Linux-only, single-binary deployment model.
