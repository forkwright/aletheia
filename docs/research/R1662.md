# R1662: Evaluate Qwen3.5-397B-A17B as Local Aletheia Model

**Date:** 2026-03-19
**Author:** Research agent
**Status:** Final
**Closes:** #1662

---

## Executive Summary

Qwen3.5-397B-A17B is a compelling MoE model that substantially outperforms Claude Sonnet 4.5 on tool-use benchmarks (BFCL-V4: 72.9 vs 54.8) and instruction following (IFBench: 76.5 vs 58.0), while costing ~8× less per token via cloud API. However, integrating it into aletheia carries non-trivial engineering cost: the `hermeneus` crate is Anthropic-native, and moving to any non-Anthropic model means implementing an OpenAI-compatible provider, losing prompt caching, and losing extended thinking. The "local" framing is misleading — the 397B model requires 200+ GB VRAM, ruling out single-node consumer deployment. Via cloud API ($0.39–$0.60/M input), it is genuinely interesting as a cost-efficient alternative to Claude Sonnet 4.6, but not as a self-hosted model.

**Recommendation: Investigate further.** The model's quality is real and the API pricing makes a Claude Sonnet 4.6 → Qwen3.5-397B-A17B substitution potentially worthwhile. The prerequisite is first building OpenAI-compatible provider support in `hermeneus` (tracked separately). Without that, there is nothing to evaluate against aletheia's actual prompts.

## 1. Model Identity

The name "Qwen 3.5-397B-A17B" in the issue is correct. It refers to **Qwen3.5-397B-A17B**, the flagship of Alibaba's Qwen3.5 series, released **February 16, 2026**. It is a vision-language MoE model:

| Property | Value |
|---|---|
| Total parameters | 397B |
| Active parameters per token | 17B (10 routed + 1 shared expert) |
| Total experts | 512 |
| Transformer layers | 60 |
| Context window (native) | 262,144 tokens |
| Context window (YaRN) | ~1,010,000 tokens |
| Architecture | Hybrid Gated DeltaNet + sparse MoE |
| Modalities | Text, image, video (input) |
| Languages | 201 |
| License | Apache 2.0 |
| Training data | ~36T token corpus (extending Qwen3 base) |

The hybrid architecture (3:1 ratio of linear-to-full-attention layers) is the key to its speed: Alibaba reports **8.6× faster decoding at 32K context** and **19× faster at 256K context** compared to Qwen3-Max.

The BFCL-V4 score of 72.9 is the best available public benchmark for JSON-schema-style function calling, which is the format aletheia uses (`ToolDefinition` with `input_schema: serde_json::Value`). Qwen3.5 leads every measured Claude variant on this benchmark.

The MCPMark benchmark (real-world MCP tool use) shows all models in the 24–52% pass@1 range, indicating the harder multi-step regime is far from solved.

## 3. Aletheia Integration Analysis

### 3.1 Current architecture: Claude-exclusive

`hermeneus` (`crates/hermeneus/`) is an Anthropic-native client (~600 LOC HTTP client, ~7K LOC total). The `LlmProvider` trait (`provider.rs:25`) is generic enough to support other providers, and `ProviderConfig` (`provider.rs:81`) already includes `provider_type: String` with a comment noting `"anthropic", "openai", "ollama"` as intended values. A Qwen integration path exists in the architecture.

What would be required:
- New module `hermeneus/src/openai/` implementing `LlmProvider` for OpenAI-compatible APIs (Qwen uses this wire format)
- Model registration in `SUPPORTED_MODELS` equivalent for the new provider
- Config schema update in `taxis` to express `provider_type = "openai"` with `base_url`

Estimated scope: **300–500 LOC** in `hermeneus`. No changes to `nous`, `organon`, or tool definitions required — tool JSON schema is the same format.

### 3.2 Features lost with any non-Anthropic model

| Feature | Aletheia support | Non-Anthropic |
|---|---|---|
| Prompt caching (`cache_system`, `cache_tools`, `cache_turns`) | Yes — reduces cost on long sessions | Not supported |
| Extended thinking (`thinking_enabled`) | Yes — configurable per agent | Not supported |
| OAuth credential management | Yes — auto-refresh | Not applicable |
| Server-side tools (`web_search`, `code_execution`) | Yes | Not applicable |
| Anthropic citations | Optional | Not applicable |

Prompt caching is a real cost driver for aletheia's system prompt stack. A typical session might have 40K tokens of system context (SOUL.md + TOOLS.md + GOALS.md + MEMORY.md + AGENTS.md). With caching, the system prompt costs are paid once then cached; without caching every turn pays full input price. This narrows the effective cost advantage of switching.

### 3.3 System prompt compatibility

Aletheia's bootstrap assembles system prompts from multiple workspace files with a priority ordering (`nous/src/bootstrap/`). The assembled prompt is rich structured Markdown with sections including SOUL, GOALS, TOOLS, MEMORY, and IDENTITY.

Qwen3.5-397B-A17B scores 92.6 on IFEval and 76.5 on IFBench — among the strongest instruction-following numbers available. No systematic testing against aletheia's actual prompts has been performed, but the model's characteristics are favorable for:
- Multi-section Markdown instructions
- Persona adherence (SOUL.md)
- Tool inventory comprehension (TOOLS.md)
- Memory integration (MEMORY.md)

Testing with aletheia's real prompts is the missing evidence. This requires a functioning OpenAI-compatible provider first.

## 4. Local Deployment Reality Check

The issue asks about Qwen3.5-397B-A17B as a **local** model. The hardware requirements make this impractical for most aletheia operators:

| Quantization | Disk | VRAM / RAM | Notes |
|---|---|---|---|
| BF16 full | ~794 GB | ~800 GB VRAM | Multi-node H100 cluster |
| FP8 | ~397 GB | ~400 GB VRAM | 5× H100 80GB |
| Q4_K_XL (GGUF) | ~214 GB | ~256 GB | M3 Ultra (512GB) or large RAM server |
| Q3_K_XL (GGUF) | ~192 GB | ~192 GB | M3 Ultra (192GB, borderline) |

Inference speed at Q4 on consumer hardware:
- **8× H100 cluster**: ~45 tok/s (FP8, vLLM)
- **M3 Ultra 192GB**: ~25–30 tok/s (estimated, Q3)
- **1× RTX 4090 + 256GB DDR5 RAM**: 25+ tok/s (Q4, MoE offload) — viable but requires tuning

**Key limitation:** Ollama, the standard local LLM deployment path for most users, does not support Qwen3.5-397B-A17B as of March 2026 due to incompatibility with the vision encoder's mmproj files. Smaller Qwen3.5 models (9B, 27B, 35B) do run on Ollama.

For practical "local" deployment of a Qwen3.5 MoE, the most accessible option is:
1. Use the cloud API (Alibaba Cloud, OpenRouter, Together AI) — not local but zero-infra
2. Deploy Qwen3.5-35B-A3B locally (single RTX 4090, 24GB VRAM, BFCL 67.3) — genuinely local but lower capability

### Smaller Qwen3.5 alternatives for true local use

| Model | BFCL | SWE-bench | Context | VRAM (Q4) | Practical hardware |
|---|---|---|---|---|---|
| Qwen3.5-35B-A3B | 67.3 | 69.2 | 262K | ~24 GB | Single RTX 4090 |
| Qwen3.5-27B (dense) | 68.5 | **72.4** | 262K | ~20 GB | Single RTX 4090 |
| Qwen3.5-122B-A10B | 72.2 | 72.0 | 262K | ~81 GB | 2× RTX 6000 Ada or M2 Ultra |

The **27B dense model** is particularly interesting for aletheia: 72.4 SWE-bench and 68.5 BFCL on a single RTX 4090, matching or exceeding Claude Sonnet 4.5 on those benchmarks.

## 5. Cost Analysis

### 5.1 Cloud API pricing comparison

| Model | Input ($/M tok) | Output ($/M tok) | Relative to Claude Sonnet 4.6 input |
|---|---|---|---|
| Qwen3.5-397B-A17B (OpenRouter) | **$0.39** | **$2.34** | ~8× cheaper |
| Qwen3.5-397B-A17B (Alibaba/Together) | $0.60 | $3.60 | ~5× cheaper |
| Qwen3.5-122B-A10B | ~$0.10 | ~$0.40 | ~30× cheaper |
| Qwen3.5-35B-A3B (API) | ~$0.06 | ~$0.24 | ~50× cheaper |
| **Claude Haiku 4.5** | $0.80 | $4.00 | — |
| **Claude Sonnet 4.6** | $3.00 | $15.00 | 1× (baseline) |
| **Claude Opus 4.6** | $15.00 | $75.00 | 5× more expensive |

### 5.2 Effective cost with aletheia prompt caching

Aletheia caches system prompts (`cache_system`, `cache_tools`). At 40K tokens of system context, prompt caching can reduce effective input cost by 60–80% on long sessions (Anthropic charges ~10% of input price for cache reads). Without caching, Qwen3.5's $0.39/M looks great; with caching, Claude Sonnet's effective cost on repeat turns drops to ~$0.30–$0.60/M input — more competitive.

The cost advantage of Qwen3.5 via API is real but less dramatic once accounting for caching. For short or single-turn sessions where caching provides no benefit, Qwen3.5 at $0.39/M is a clear winner.

### 5.3 Self-hosted cost

At sufficient scale (>5M tokens/month output), self-hosting Qwen3.5 on an M3 Ultra or GPU server becomes cost-effective versus Claude API. The hardware investment ($5K–$15K for M3 Ultra) breaks even vs Claude Sonnet 4.6 at roughly 3–6 months of heavy use. For privacy-sensitive deployments, self-hosting eliminates all per-token costs.

## 6. Risks and Unknowns

1. **No aletheia-specific test results.** All benchmark data is from public evaluations. Aletheia's actual system prompts (especially the tool inventory in TOOLS.md, and cross-agent AGENTS.md) have not been tested against Qwen3.5. Instruction-following benchmarks are promising but not a substitute for direct testing.

2. **Multi-step tool loop reliability is unproven.** BFCL measures single-turn tool selection. Aletheia's tool loop runs up to 50 iterations with error recovery. MCPMark (real-world multi-step) shows all models at 24–52% pass@1 — the operational reliability gap vs Claude at this task tier is unknown.

3. **Thinking mode / fast mode sampling differences.** Qwen3.5 requires different sampling parameters (`temperature`, `top_p`) for thinking vs non-thinking modes. This affects how well default configs will work. Aletheia would need mode-aware defaults.

4. **No prompt caching support.** Aletheia's system prompt stack is designed for Anthropic's caching primitives. Without them, effective cost increases on multi-turn sessions.

5. **Vision capability untested.** Qwen3.5-397B-A17B is a VLM. Aletheia does not currently use vision, but this is future-relevant.

6. **Ollama incompatibility at 397B.** For local deployment specifically, the standard toolchain doesn't work at full scale.

## 7. Recommendation

**Investigate further.** This is not a skip — the quality evidence is genuinely compelling. But current aletheia architecture cannot use Qwen3.5 without engineering work first, and no aletheia-specific empirical data exists.

### Prerequisite (blocks all downstream evaluation)

Build OpenAI-compatible provider support in `hermeneus` (`crates/hermeneus/src/openai/`). This unlocks Qwen3.5 and all other OpenAI-compatible providers (Groq, Together, Fireworks, local vLLM/llama.cpp). ~300–500 LOC.

### Recommended evaluation sequence (once provider exists)

1. **Qwen3.5-35B-A3B or 27B** via cloud API — cheapest path to gather aletheia-specific signal. Test with full system prompt stack and tool definitions. Focus on: tool call accuracy across all 33 built-in tools, multi-turn tool loop completion, SOUL.md persona adherence.

2. **Qwen3.5-397B-A17B** via cloud API (OpenRouter at $0.39/M input) — test head-to-head against Claude Sonnet 4.6 on same prompts.

3. **Local 397B only if** the org has hardware capable of running it (M3 Ultra 192GB+ or multi-GPU server). Not worth evaluating otherwise.

### Decision criteria for adoption

| Criterion | Target | Risk if missed |
|---|---|---|
| Tool call accuracy (BFCL-style, aletheia tools) | ≥ Claude Sonnet 4.6 | Tool loop failures, user-visible errors |
| Multi-step tool loop completion rate | ≥ 90% success on test corpus | Broken agent sessions |
| SOUL.md persona adherence | Qualitatively indistinguishable | Character drift, user trust issues |
| TOOLS.md comprehension | ≤ 5% incorrect tool selections | Wrong tool invocations |
| Effective cost (with/without caching) | ≥ 20% savings vs Claude Sonnet 4.6 | Insufficient migration motivation |

### Bottom line

If the OpenAI-compatible provider is built and testing yields ≥ parity with Claude Sonnet 4.6 on aletheia's actual workloads, **Qwen3.5-397B-A17B via cloud API** is a strong candidate for the primary model, with Qwen3.5-35B-A3B or 27B as the self-hosted/privacy-first option. The model's instruction-following and tool-use benchmarks are the best available for open-weight models, and the pricing gap is significant.
