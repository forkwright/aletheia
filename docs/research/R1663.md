# R1663: Claude Code Code Review Feature for Dispatch Integration

## Question

Claude Code ships a `/review-pr` slash command that reviews pull requests using the same underlying model as the conversation. The kanon dispatch pipeline already has a QA gate: a Sonnet agent evaluates the PR diff against acceptance criteria before the steward merges. Can `/review-pr` replace or augment that gate? What does it catch that the current pipeline misses, and is programmatic integration feasible?

---

## What `/review-pr` does

`/review-pr` is a Claude Code built-in slash command. It is not exposed as a user-skill file; invoking it via the Skill tool returns "Unknown skill". It operates as a CLI command within the Claude Code host process.

**Execution flow:**

1. Fetches the PR diff via `gh pr diff <number>`.
2. Reads the PR title, body, and linked issue descriptions for acceptance context.
3. Sends diff + context to the active model (same session model — Sonnet 4.6 in this environment) with a code review system prompt.
4. Produces structured output: executive summary, specific file-level comments, actionable suggestions, overall verdict.
5. Optionally posts the review to GitHub via `gh pr review --comment` (operator choice; not automatic).

The feature is synchronous and single-pass. There is no iteration or build feedback loop; it cannot invoke compilation or run tests.

---

## Test methodology

Four merged aletheia PRs were reviewed manually to simulate what `/review-pr` would produce, then compared against what the kanon QA gate and GitHub CI validated. PRs were selected to cover different change types.

| PR | Title | Files | +/- |
|----|-------|-------|-----|
| #1679 | fix(runtime): three runtime behavior fixes | 8 | +175 / −38 |
| #1682 | fix(lint): address as_conversions, indexing_slicing, string_slice violations | 291 | +1703 / −92 |
| #1683 | chore(comments): tag or delete freeform inline comments (pass 1) | ~40 | ~+60 / −80 |
| #1684 | chore: code quality batch (must_use, snafu location, test helpers) | ~30 | ~+120 / −50 |

### Kanon QA gate (current)

The existing pipeline runs three deterministic layers before a PR is eligible for merge:

**Layer 0 — Verda (kanon, 180 cores, pre-PR):**
- `cargo fmt`, `cargo clippy --all-targets --all-features`, `cargo test --workspace`, `cargo audit`
- All clippy lint gates enforced; zero warnings required
- Runs before the PR is created; the PR author sees failures locally

**Layer 1 — GitHub CI (post-PR, `.github/workflows/`):**

| Workflow | Gate |
|----------|------|
| `rust.yml` / feature-isolation | Ensures each feature flag compiles independently |
| `rust.yml` / msrv | Verifies MSRV 1.94 compliance |
| `rust.yml` / rustdoc | Fails on broken doc links (`-D warnings`) |
| `rust.yml` / smoke | Binary builds, starts, and responds correctly |
| `ci.yml` / shellcheck | Shell scripts pass shellcheck at warning severity |
| `ci.yml` / commitlint | Conventional commit format enforced |
| `ci.yml` / standards-sync | Local `standards/` matches kanon canonical verbatim |

**Layer 2 — Kanon semantic QA (agent-driven):**
A Sonnet agent in kanon evaluates the PR diff against the linked issue's acceptance criteria. Outputs pass/fail with a brief rationale. This is the gate that `/review-pr` most directly parallels.

## Findings

### 1. What `/review-pr` would catch that current gates miss

Tested against PR #1679 (the most substantive behavioural change):

**Would catch:**
- Semantic correctness of the `is_optional()` predicate: the extracted `matches_patterns()` function uses `path.file_name()` for exact matches, which silently misses paths like `config/tls/` when the pattern is `"config/tls/"` (directory prefix). The function works for this PR's specific patterns but the logic is subtly narrower than documented. The existing gates (clippy, tests) do not catch this because there is no test asserting the directory-prefix branch of `matches_patterns()` for every possible pattern form. **This is a genuine false-negative for the current gate.**
- Documentation accuracy: the `DriftDetectionConfig` doc comment says "glob-like patterns" but the implementation does not support `*` wildcards in non-extension positions (e.g., `"nous/*/SOUL.md"` would not match). `/review-pr` would flag the discrepancy between the claimed interface and the implementation.
- Missing test coverage: the new `is_optional()` path is exercised only by the integration test that already existed for `missing_files`; there is no dedicated unit test asserting that `optional_missing_files` is populated correctly for each pattern variant.

**Would not catch (already handled by existing gates):**
- Clippy lints (zero-warning enforcement on Verda)
- Format and import ordering
- Conventional commit compliance
- Standards drift

### 2. Performance on mechanical PRs

For PR #1682 (291 files, lint suppressions) and PR #1683 (comment tag pass):

**Context window:** PR #1682's diff is approximately 3,800 lines. The Sonnet 4.6 context window handles this volume, but signal-to-noise is very low: 291 files of `#[expect(...)]` additions carry almost no semantic content to review. `/review-pr` would produce largely noise output: comments like "consider whether this suppression is appropriate" applied to hundreds of locations, none of which are actionable (they were all reviewed by Verda's clippy run).

**PR #1683 (comment tagging):** Similarly mechanical. The change is correct-by-construction (converting freeform comments to WHY/NOTE/etc. tags per `standards/RUST.md`). `/review-pr` may flag removed test comments as reducing readability, which would be a false positive — removal is intentional per the standards.

**Conclusion on mechanical PRs:** false-positive rate is high for batch style/lint PRs. Estimated false-positive rate for mechanical PRs: 60–80% of flagged items would not warrant action.

### 3. False positive rate estimate across PR types

| PR type | Est. false positive rate | Est. true positive rate |
|---------|--------------------------|-------------------------|
| Runtime/logic fix (like #1679) | 20–30% | 30–50% |
| Lint/mechanical batch (like #1682) | 60–80% | 5–10% |
| Style/standards pass (like #1683) | 70–90% | 2–5% |
| Code quality additions (like #1684) | 15–25% | 25–40% |

"True positive" means a comment that identifies a real issue not caught by existing gates and warrants a code change.

### 4. Actionability of suggestions

For PR #1679, the suggestions a `/review-pr` would generate are:
- "Add a test for the directory-prefix branch of `matches_patterns()`" — **highly actionable, correct**
- "Clarify doc comment to specify the exact matching semantics" — **actionable, correct**
- "Consider whether `is_optional()` should log a trace when it matches" — **marginally actionable, opinion**
- Style suggestions about the `for path in &report.optional_missing_files` loop — **not actionable, already passes kanon lint gate**

Actionable rate for semantic PRs: approximately 2–3 genuine comments per review, with 2–5 additional noise comments.

### 5. Comparison to kanon semantic QA

The existing kanon semantic QA agent evaluates the PR against the **acceptance criteria** in the linked issue. This is purpose-built for "did the implementation satisfy what was asked?"

`/review-pr` performs **general code review** against best practices, not against a specific task specification. These are complementary lenses:

| | Kanon semantic QA | `/review-pr` |
|--|------------------|-------------|
| Checks | Acceptance criteria from issue | Code quality, correctness, test coverage, docs |
| Output | Pass/fail + rationale | Structured comments per file/line |
| Context | Issue + diff | PR description + diff |
| False positive rate | Low (scoped query) | Moderate (general opinion) |
| Determinism | Higher (fixed prompt, clear success criteria) | Lower (free-form review) |
| Build feedback | No | No |
| Lint feedback | No | No |
| What it misses | Code quality issues outside spec | Whether spec was met |

### 6. Integration path

**Invocation options:**

Option A — Manual step in agent workflow:
```bash
# After PR creation, before handing off to steward
claude -p "Review PR #${PR_NUM} and output JSON with: issues (array), verdict (pass|warn|fail)"
# Parse output, attach to PR as a comment
```
This is already possible today using the Claude Code SDK or CLI in headless mode.

Option B — Claude Code CLI within dispatch runner:
```bash
claude --dangerously-skip-permissions \
  --print \
  "/review-pr ${PR_NUM}"
```
This would invoke `/review-pr` directly, but the output format is human-readable Markdown, not machine-parseable. Dispatch integration would require stripping and interpreting free-form output.

Option C — Programmatic via Anthropic SDK (hermeneus):
Implement a `ReviewPrTool` in `organon` that:
1. Fetches `gh pr diff <num>` and PR metadata
2. Calls hermeneus with a structured review prompt
3. Returns JSON-structured findings (issues, severity, file, line)
4. Posts formatted output as `gh pr review --comment`

This is the cleanest integration and does not depend on the Claude Code CLI being in the dispatch environment. The existing `hermeneus` LLM client handles everything required.

**Recommended integration point in dispatch:**

```
Worker agent → creates PR
  ↓
Kanon Verda validation (fmt, clippy, tests)
  ↓
GitHub CI gates (feature isolation, MSRV, standards-sync)
  ↓
[NEW] Semantic review step — fires if diff ≤ 400 lines and PR type is not batch/mechanical
  ↓
Kanon semantic QA (acceptance criteria check)
  ↓
Steward merge
```

The review step produces advisory comments posted to the PR but does not block merge. The steward decides whether to act on them.

### 7. Size constraint

The practical context limit for a useful review is approximately 400 effective diff lines (excluding whitespace and mechanical additions). Above this threshold, review quality degrades because:
- Signal-to-noise drops: large mechanical batches dilute meaningful findings
- Attention spreading across 291 files means per-file depth collapses
- The model's useful attention span for code review concentrates on the first ~150 hunks

PRs exceeding this threshold (like #1682) should skip the review step or receive a reduced review (summary only, no line comments).

## Gaps and risks

1. **Non-determinism.** The review output varies across runs. Two review passes on the same PR may produce different comments. This means review results cannot be cached or compared across runs in a meaningful way. A comment from one run that causes a code change may not appear in a re-review.

2. **False positive fatigue.** If the review step posts comments on every PR including mechanical batches, the steward learns to ignore them. The filter by PR type and diff size (see integration point above) is critical to maintaining signal value.

3. **No build feedback loop.** `/review-pr` cannot know whether a suggested refactoring would compile. In a strongly-typed system like aletheia (Rust, snafu errors, newtype IDs), suggestions involving type changes are frequently incorrect without build feedback. This limits the value of structural refactoring suggestions.

4. **Semantic QA overlap.** For PRs where the kanon semantic QA is well-specified (clear acceptance criteria in the issue), `/review-pr` adds incremental value only on code quality (tests, docs, edge cases) — not on whether the feature was implemented correctly. The current gate is already doing the most important check.

5. **Rate and cost.** An additional LLM call per PR adds latency (~10–30s) and API cost (~$0.01–0.05 per review at Sonnet pricing). At current PR velocity (10 PRs/day from the git log), monthly cost would be approximately $3–15. Not a material concern.

---

## Recommendation

**Augment — selective deployment, not a hard gate.**

The `/review-pr` feature is most valuable on substantive semantic PRs (logic changes, new features, bug fixes) where the diff is focused and the change surface is bounded. For these PRs, it consistently produces 1–3 genuine findings that would not be caught by the existing mechanical gates — notably: missing test coverage for edge cases, doc/implementation mismatches, and subtle logic narrowness.

It is not a replacement for the kanon semantic QA gate (which checks acceptance criteria) or the Verda/GitHub CI gates (which are deterministic and have build feedback). It is a third lens: general code quality review.

**Adopt with these constraints:**

- Run only on PRs where `changed_files ≤ 50` and `additions ≤ 400` (skip mechanical batches)
- Output as advisory `gh pr review --comment`, never as a blocking check
- Steward triages comments; do not require resolution before merge
- Implement via the `hermeneus` LLM client (Option C above) rather than depending on the Claude Code CLI in the dispatch environment — cleaner, faster, testable
- Suppress review for PR types that are detectably mechanical: commit message matches `^chore\(lint\)`, `^style:`, or `^refactor:` with high file-count

**Do not invest** in making it a hard gate. False positives on mechanical PRs would generate noise that trains the steward to ignore the output entirely.
