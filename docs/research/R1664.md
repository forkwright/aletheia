# R1664: Agency-Agents Repository — Applicable Patterns for Aletheia

**Date:** 2026-03-19
**Author:** Research agent
**Status:** Final

---

## Executive Summary

`msitarzewski/agency-agents` (GitHub, MIT, 54.9k stars) is a curated library of 144+ Markdown-defined AI agent personalities for use with Claude Code, Cursor, Aider, and related tools. It is structurally the opposite of Aletheia: where Aletheia is a compiled Rust runtime with strongly typed pipelines, `agency-agents` is a collection of plain-text prompt documents with no executable code. Despite that gap, several concrete patterns are transferable.

The most actionable findings are:

1. **Structured YAML frontmatter in agent definitions** — a lightweight, parseable identity contract that Aletheia's SOUL.md currently lacks.
2. **Measurable success metrics embedded in agent prompts** — quantitative targets (latency, uptime, accuracy) that agents can self-check.
3. **Deliverable-first prompt framing** — prompts define outputs before processes, reducing hallucinated narration.
4. **Explicit delegation routing rules** — typed delegation tables (by role, model tier, and task type) already present in Aletheia's `_template/AGENTS.md`, but not consistently in individual agent SOUL.md files.
5. **Phase-gated orchestration with max-retry escalation** — the AgentsOrchestrator's 3-attempt cap before escalation is a concrete guard pattern worth adopting in Aletheia's dianoia planning crate.

## Aletheia Current Architecture Overview

Aletheia is a Rust workspace of 18 crates (`crates/`) producing a single self-hosted binary.

### Crate layers

| Layer | Crates | Role |
|-------|--------|------|
| Leaf | `koina` | Errors, tracing, fs utils |
| Low | `taxis`, `hermeneus`, `mneme`, `organon`, `agora`, `melete` | Config, LLM client, memory+Datalog+HNSW, tool registry, channels, distillation |
| Mid | `nous` | Agent pipeline actor model |
| High | `pylon`, `diaporeia` | HTTP gateway, MCP server |
| Top | `aletheia` | Binary entrypoint |

### Agent pipeline (per turn)

```
Guard → Bootstrap → Skills → Recall → History → Execute → Finalize
```

Each stage is an instrumented async function in `crates/nous/src/pipeline/stages.rs`. The actor model (`NousActor`) processes turns sequentially, spawning each pipeline into a separate Tokio task for panic isolation.

### Agent identity model

Each agent (nous) has a workspace directory (`instance/nous/<id>/`) with up to 9 Markdown files:

| File | Priority | Source |
|------|----------|--------|
| SOUL.md | 1 (Required) | Hand-written |
| USER.md | 2 | Hand-written |
| AGENTS.md | 3 | Hand-written |
| GOALS.md | 4 | Hand-written |
| TOOLS.md | 5 | Auto-generated |
| MEMORY.md | 6 | Nous-maintained |
| IDENTITY.md | 7 | Hand-written |
| PROSOCHE.md | 8 | Auto-generated |
| CONTEXT.md | 9 | Runtime-written |

The `BootstrapAssembler` packs these into a single system prompt within a token budget (default 40k tokens), truncating flexible sections and dropping optional ones under pressure.

### Memory

`mneme` provides an embedded Datalog+HNSW engine. Recall uses hybrid vector + BM25 retrieval with MMR diversity. Distillation (`melete`) triggers at >120k context tokens, >150 messages, or 7+ day stale sessions.

### Multi-agent coordination

`CrossNousRouter` provides typed async messaging (`send`, `ask`/reply) with cycle detection (directed graph) and delivery audit logging. Agents delegate via `sessions_send`, `sessions_ask`, and `sessions_spawn` tools. `dianoia` handles multi-phase planning orchestration.

### Tool execution

`organon` provides 33 built-in tools across categories: workspace (read/write/edit/exec), memory, communication, agent coordination, planning, and research. All implement the `ToolExecutor` trait. Linux sandbox uses Landlock + seccomp.

## Agency-Agents Repository Overview

**URL:** https://github.com/msitarzewski/agency-agents
**License:** MIT
**Stars:** 54.9k | **Forks:** 8.2k
**Commits:** 210

A collection of Markdown files, each defining one AI agent personality. Organized into 18 domain divisions:

```
engineering/     (23 agents)    design/          (8 agents)
marketing/       (26 agents)    sales/           (8 agents)
product/         (5 agents)     project-management/ (6 agents)
testing/         (8 agents)     support/         (6 agents)
spatial-computing/ (6 agents)   game-development/ (20+ agents)
academic/        (5 agents)     specialized/     (27 agents)
```

Support infrastructure: `scripts/install.sh` (multi-tool installer), `scripts/convert.sh` (converts to Cursor/Aider/Windsurf formats).

### How agents are defined

Each file begins with YAML frontmatter, followed by structured Markdown sections:

```yaml
---
name: AI Engineer
description: Expert AI/ML engineer specializing in machine learning model development...
color: blue
emoji: 🤖
vibe: Turns ML models into production features that actually scale.
---
```

Followed by required sections:

- **Identity & Memory** — voice, personality traits, domain background
- **Communication Style** — response format, terminology preferences
- **Critical Rules** — hard constraints the agent must not violate
- **Core Mission** — specialized focus areas
- **Technical Deliverables** — concrete outputs with code examples
- **Workflow Process** — step-by-step execution patterns
- **Success Metrics** — quantitative performance targets
- **Advanced Capabilities** — optional extended skills

### Multi-agent coordination (AgentsOrchestrator)

The specialized `agents-orchestrator.md` defines a phase-based pipeline:

1. PM analysis (task list creation)
2. Technical architecture foundation
3. Development-QA loops (per-task, max 3 retries before escalation)
4. Final integration and reality-checking

Context handoffs carry "complete context and specific instructions" referencing exact file locations and prior deliverables. The orchestrator tracks retry counts, phase progress, and QA evidence per task.

## Findings

### Pattern 1: YAML frontmatter as a parseable identity contract

**Description:** Every `agency-agents` file begins with YAML frontmatter containing `name`, `description`, `color`, `emoji`, and `vibe`. This is machine-readable, tooling-friendly, and provides a compact structured summary separate from the prose body.

**Evidence:**

```yaml
---
name: Backend Architect
description: Expert in scalable system design, database architecture, API development...
color: blue
emoji: 🏗️
vibe: Builds systems that survive contact with reality.
---
```

Aletheia's current equivalent is `IDENTITY.md`, which the docs describe as "Minimal metadata: name, emoji. Parsed as frontmatter." (`docs/WORKSPACE_FILES.md`, line 17). However, SOUL.md — the primary identity document — is unstructured prose. The `description` and `vibe` fields have no counterpart.

**Recommendation: Adopt (partial)**

Add an optional `description` field to `IDENTITY.md` frontmatter. This gives the `NousManager` and API endpoints (`GET /api/v1/nous/{id}`) a structured one-line description without modifying SOUL.md's prose contract. The `BootstrapAssembler` already parses IDENTITY.md frontmatter; adding `description` requires only extending the struct and the API response type.

Do not adopt `color` or `vibe` at the runtime layer — these are UI/display concerns. They could be added as optional IDENTITY.md frontmatter fields consumed by the TUI.

### Pattern 2: Measurable success metrics embedded in agent prompts

**Description:** Every `agency-agents` agent defines quantitative targets in a "Success Metrics" section. These are not aspirational — they are specific enough that the agent (or an observer) can evaluate them:

- Backend Architect: API p95 < 200ms, 99.9% uptime, DB queries avg < 100ms, zero critical CVEs, 10x traffic headroom
- AI Engineer: Model accuracy > 85%, inference latency < 100ms, uptime > 99.5%, 20%+ engagement improvement

**Evidence from `engineering/engineering-ai-engineer.md`:**

```
Success Metrics:
- Model accuracy targets of 85%+
- Inference latency under 100ms for real-time applications
- Service uptime exceeding 99.5%
- Typical user engagement improvements of 20%+
```

Aletheia's SOUL.md template (`instance.example/nous/_template/SOUL.md`) has three open sections: "What I am", "How I work", "What I don't do". There is no success metrics section. The `GOALS.md` file holds goals, but goals and metrics are distinct: goals are tasks, metrics are evaluation criteria.

**Recommendation: Adopt**

Add a "Success metrics" section to the SOUL.md template. For each agent, operators define 3–5 quantitative or observable criteria. During the prosoche heartbeat, the agent can evaluate itself against these criteria — a natural extension to the existing `PROSOCHE.md` checklist system documented in `docs/PROSOCHE.md`.

This is purely a prompt-engineering change: update `instance.example/nous/_template/SOUL.md` and `docs/WORKSPACE_FILES.md`. No runtime code changes required.

### Pattern 3: Deliverable-first prompt framing

**Description:** `agency-agents` prompts lead with "Technical Deliverables" — specific output artifacts — before describing process or personality. The Backend Architect's deliverables section lists: system architecture specifications, database schemas with indexing examples, API design specs with security middleware. The MCP Builder's deliverables include TypeScript server templates with Zod validation.

This ordering — output before process — reduces the model's tendency to narrate actions rather than produce results. It anchors the agent's response to a concrete artifact type.

**Evidence:** The contributing guide (`CONTRIBUTING.md`) formalizes this as a required section: "📋 Your Technical Deliverables" appears before "🔄 Your Workflow Process" in the template structure.

Aletheia's `_template/AGENTS.md` partially addresses this through delegation rules ("Check `status`/`confidence`. High confidence + routine -> integrate.") but the SOUL.md template has no deliverable specification.

**Recommendation: Adopt**

Add a "Deliverables" subsection to SOUL.md's "How I work" section in the template. Operators enumerate 2–4 primary output types for each nous. This is a documentation and prompt-engineering change only.

### Pattern 4: Delegation routing table with model tier assignment

**Description:** The `agency-agents` orchestrator and Aletheia's own `_template/AGENTS.md` both define delegation routing. However, `agency-agents` makes model tier selection explicit at the routing-table level:

From `instance.example/nous/_template/AGENTS.md`:
```
| Role      | Model  | Use For |
|-----------|--------|---------|
| coder     | Sonnet | Code, edits, migrations, builds, lint/type fixes |
| reviewer  | Sonnet | Diff/PR review, bugs, style |
| researcher| Sonnet | Web research, API docs, information gathering |
| explorer  | Haiku  | Read-only codebase investigation - grep, trace, find |
| runner    | Haiku  | Execute commands, run tests, health checks, logs |
```

This is already present in the template. The gap is that individual agent SOUL.md files are not required to reproduce or override this table — a newly scaffolded agent inherits the template's AGENTS.md but operators may omit it.

The `agency-agents` pattern goes further: the AgentsOrchestrator dynamically selects agents by domain ("frontend developers for UI work, backend architects for server logic, senior developers for premium implementations") based on task type, not just a static table.

**Recommendation: Investigate further**

The static delegation table in the template is correct. The gap is that `dianoia` (planning crate) and `sessions_spawn` do not currently perform task-type-based agent selection — they spawn by explicit ID. Adding a task-type → agent-id routing registry would let the orchestrator pattern from `agency-agents` be applied in Aletheia's `dianoia` crate. This requires runtime changes and should be evaluated against M5/M6 roadmap items.

### Pattern 5: Phase-gated orchestration with bounded retry and escalation

**Description:** The `agents-orchestrator.md` enforces a strict per-task retry budget (max 3 attempts) before marking a task blocked and continuing. Failed tasks do not loop indefinitely. The orchestrator tracks: current phase, task completion percentage, retry counts per task, and QA evidence.

```
Task fails → retry (max 3 total) → if still failing: mark blocked, advance pipeline
Agent spawn fails → retry (max 2 attempts) → error reported
```

**Evidence:**

> "Failed tasks loop maximum 3 times before escalation. The system defaults to 'NEEDS WORK' unless overwhelming evidence proves production readiness."

Aletheia's pipeline guard stage (`crates/nous/src/pipeline/stages.rs`, `run_guard_stage`) detects loops via `LoopDetected` but this operates at the turn level, not the task level. `dianoia` (`crates/dianoia`) handles multi-phase planning, but retry capping per task step is not surfaced in the architecture documentation.

The `NousActor` enters degraded mode after 5 panics in a 10-minute window (from `crates/nous/CLAUDE.md`), which is an analogous bounded-retry pattern at the actor level.

**Recommendation: Adopt**

Extend the `dianoia` planner to track per-step attempt counts and enforce a configurable max-retries cap (default 3) before emitting a `StepBlocked` event. The `plan_step_fail` built-in tool already signals failure; adding an attempt counter to the plan step state type would be the minimal change. The `LoopDetected` guard could also be extended to emit structured retry metadata rather than a binary error.

This is a targeted enhancement to existing infrastructure.

### Pattern 6: "Critical Rules" as hard constraints distinct from identity

**Description:** `agency-agents` separates behavioral constraints into a dedicated "Critical Rules You Must Follow" section, distinct from personality ("Identity") and capability ("Core Mission"). Examples from Backend Architect:

- "Never skip authentication and authorization checks"
- "Always implement defense in depth strategies"
- "Apply principle of least privilege for all services"

These are categorical prohibitions, not preferences. The separation ensures constraints survive prompt truncation and cannot be overridden by conversational context.

**Evidence:** The CONTRIBUTING.md template lists "🚨 Critical Rules You Must Follow" as a required section between Identity and Core Mission.

Aletheia's SOUL.md template has "What I don't do" which is semantically similar but less emphatic. The bootstrap assembler applies priority ordering (SOUL.md is Required, never dropped) but does not distinguish between preference-level and constraint-level content within SOUL.md.

**Recommendation: Adopt (template only)**

Add an explicit "Hard constraints" subsection to the SOUL.md template between "What I am" and "How I work". Document this in `docs/WORKSPACE_FILES.md` as the location for absolute prohibitions. This is a pure prompt-engineering change with no runtime impact.

### Pattern 7: Workflow Architect's observable state layers for each workflow step

**Description:** The `specialized-workflow-architect.md` agent defines parallel state layers for every workflow step:

- Customer perception (UI state, error messages)
- Operator view (admin dashboard state)
- Database state (entity status, job metadata)
- System logs (timestamped events)

And a cleanup inventory tracking every resource created and its destruction mechanism. This "state layer" decomposition prevents invisible state bugs in complex multi-step workflows.

**Evidence:**

> "For each workflow step, the agent defines parallel state representations: customer perception, operator view, database state, system logs."

Aletheia's planning tools (`plan_create`, `plan_execute`, `plan_verify`, `plan_step_complete`, `plan_step_fail`) track step-level status but do not require multi-layer observability documentation per step.

**Recommendation: Investigate further**

This pattern is most valuable for long-running workflows with external side effects (database writes, emails, API calls). Aletheia's `dianoia` planner could optionally require an `observable_states` field per plan step. However, this adds authoring overhead. Evaluate on a case-by-case basis for agents managing stateful external workflows. Start with a WORKFLOWS.md workspace file (already present in the template directory at `instance.example/nous/_template/WORKFLOWS.md`) to document step-level observability informally before encoding it in the plan schema.

### Pattern 8: BM25 task-context extraction for skill retrieval (already in Aletheia)

**Description:** `agency-agents` agents embed "domain-specific knowledge bases" and "learning patterns" in memory sections. Aletheia already implements this more rigorously: `crates/nous/src/skills.rs` extracts a 200-character BM25 query from the current user message to retrieve relevant skills from the knowledge store, injecting them at `SectionPriority::Flexible` in the bootstrap.

**Evidence from `crates/nous/src/skills.rs`:**

```rust
const MAX_CONTEXT_CHARS: usize = 200;

pub(crate) fn extract_task_context(content: &str) -> String {
    // strips FTS-unparseable chars, truncates at word boundary
}
```

`agency-agents` has no equivalent runtime mechanism — skill retrieval is implicit (the model's weights) rather than explicit. Aletheia's implementation is superior.

**Recommendation: Skip**

This is an area where Aletheia leads. No adoption needed.

---

### Pattern 9: Multi-tool conversion infrastructure

**Description:** `agency-agents` ships `scripts/convert.sh` and `scripts/install.sh` that convert the same agent Markdown files to formats for Cursor (`.mdc`), Aider (`CONVENTIONS.md`), Windsurf (`.windsurfrules`), and others. The agent definitions are the canonical source; tooling-specific formats are derived.

Aletheia has `aletheia export-skills` and `aletheia seed-skills` CLI commands, and `diaporeia` exposes agents via MCP. The M6 roadmap lists "A2A interop."

**Evidence:**

```bash
./scripts/install.sh --tool cursor   # converts and installs for Cursor
./scripts/install.sh                  # auto-detects installed tools
```

**Recommendation: Investigate further**

`diaporeia` (MCP server) already provides one interop surface. Before M6 A2A work, evaluate whether an `aletheia export-agent --format claude-code` command would provide meaningful value. The `agency-agents` conversion script architecture (one canonical format, derived targets) aligns with Aletheia's philosophy of a single source of truth.

## Overall Recommendations

### Adopt now (prompt/template changes, no runtime code)

| Change | File | Effort |
|--------|------|--------|
| Add `description` field to IDENTITY.md frontmatter | `instance.example/nous/_template/IDENTITY.md`, `docs/WORKSPACE_FILES.md` | 1h |
| Add "Success metrics" section to SOUL.md template | `instance.example/nous/_template/SOUL.md` | 30m |
| Add "Deliverables" subsection to SOUL.md template | `instance.example/nous/_template/SOUL.md` | 30m |
| Add "Hard constraints" subsection to SOUL.md template | `instance.example/nous/_template/SOUL.md` | 30m |

These four changes require no runtime modifications. They improve every newly scaffolded agent (`aletheia add-nous`) and can be back-applied to existing agents incrementally.

### Adopt next (targeted runtime changes)

| Change | Crate | Effort |
|--------|-------|--------|
| Per-step attempt counter and max-retries cap in planner | `dianoia` | 2–4h |
| `description` field in API agent response (`GET /api/v1/nous/{id}`) | `pylon`, `nous` | 2h |

### Investigate in M5/M6 context

| Topic | Relevant Milestone |
|-------|-------------------|
| Task-type → agent-id routing in `dianoia` (dynamic orchestration) | M5 skills pipeline / M6 composable ops |
| WORKFLOWS.md step-level observability (formal schema) | M6 |
| Agent export format for Claude Code / external tools | M6 A2A interop |

### Skip

- BM25 skill retrieval (already implemented in `mneme` + `nous/src/skills.rs`, superior to `agency-agents`)
- `color`/`vibe` frontmatter at runtime layer (UI-only concern)
- Adopting `agency-agents` agent files directly (wrong abstraction level for a compiled runtime)
